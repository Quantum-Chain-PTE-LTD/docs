Every request to the Quantum Chain Custody API requires authentication via a Bearer API key. This guide covers credential format, lifecycle, and security best practices.

## API key format

Credentials consist of a **key ID** and a **secret**, separated by a colon:

```
Authorization: Bearer <key_id>:<secret>
```

Include this header in every API request:

```bash
curl -X GET "https://api.quantumchain.io/v1/tenants" \
  -H "Authorization: Bearer kid_abc123:sk_secret456"
```

## Credential lifecycle

| Stage | Description |
|-------|-------------|
| **Provisioned** | Key pair generated by the platform team or admin CLI |
| **Active** | Key is valid and accepts requests |
| **Rotated** | New key issued; old key remains valid during grace period |
| **Revoked** | Key permanently disabled; all requests return `401` |

<Warning>
Store your API secret securely. It is shown only once at creation time and cannot be retrieved later.
</Warning>

## IP allowlisting

For production deployments, restrict API access to known IP addresses. The custody API validates the source IP of each request against the tenant's allowlist.

When IP allowlisting is enabled:
- Requests from unlisted IPs receive a `403 Forbidden` response
- The allowlist is configured per tenant
- CIDR notation is supported for IP ranges

## Error codes

Authentication failures return specific error codes in the `1000` range:

| Code | Meaning |
|------|---------|
| `1000` | Missing or malformed `Authorization` header |
| `1001` | Invalid API key — key ID not found |
| `1002` | Invalid API secret — secret does not match |
| `1003` | IP address not in allowlist |

All authentication errors return HTTP `401 Unauthorized` or `403 Forbidden`:

```json
{
  "error": {
    "code": 1001,
    "message": "Invalid API key"
  }
}
```

## Best practices

<CardGroup cols={2}>
  <Card title="Rotate regularly" icon="arrows-rotate">
    Rotate API keys every 90 days. Use the rotation endpoint to issue new credentials without downtime.
  </Card>
  <Card title="Use IP allowlists" icon="shield">
    Restrict access to your known server IPs. Never call the API from client-side code.
  </Card>
  <Card title="Store secrets safely" icon="lock">
    Use a secrets manager — never commit credentials to source control or environment files.
  </Card>
  <Card title="Separate environments" icon="layer-group">
    Use different API keys for development, staging, and production.
  </Card>
</CardGroup>
